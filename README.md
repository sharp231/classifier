

# End-to-End Image Classification App (CIFAR-10)

画像認識モデルの学習から Web アプリケーションとしてのデプロイまでを、End-to-End で実装したプロジェクトです。  
ユーザーがアップロードした画像を AI が解析し、CIFAR-10 の 10 クラスに分類します。

## プロジェクト概要

機械学習モデルを作成するだけでなく、実際の Web サービスとして動作させるための設計、実装、デプロイまでを行いました。
入力画像の向きや縦横比の違いを吸収する前処理、推論結果を安定させる Test Time Augmentation、pytest と GitHub Actions による自動テストを取り入れています

- **担当領域:** 企画・要件定義 / モデル構築 / Web 実装 / テスト / デプロイ

## 技術的工夫 (Key Features)

### 1. 推論精度の安定化 (Test Time Augmentation)

単一の推論結果だけに依存せず、入力画像を複数の倍率（1.0, 0.8, 0.65）でクロップして推論し、その平均値を最終結果として採用しています。
これにより、被写体の位置や大きさの違いによる判定のぶれを抑え、より安定した分類結果を返せるようにしています。

### 2. 堅牢な前処理パイプライン

ユーザーがアップロードする画像は、撮影端末や保存形式によって向きや縦横比が異なるため、推論前に以下の前処理を行っています。

- **EXIF 自動補正:** スマートフォン写真などに含まれる回転情報を反映し、正しい向きに補正
- **Center Crop:** アスペクト比を維持したまま中央を切り抜き、単純なリサイズによる画像の歪みを防止
- **RGB 変換:** 推論モデルに入力できる形式へ統一

### 3.ロギングと運用時の可観測性

本番運用時のトラブルシューティングを行いやすくするため、print ではなく Python 標準の logging モジュールを使用しています。
ログにはタイムスタンプ、ログレベル、メッセージを含め、アップロード処理、モデル読み込み、エラー発生箇所を追跡しやすい構成にしています。
これにより、Render などの本番環境でも、エラーの発生タイミングや原因を確認しやすくしています。

### 4. 安定性を重視したアーキテクチャ

モデルの読み込みや画像アップロード処理では、アプリ全体が不安定にならないように以下を意識しています。

- **Lazy Loading:** 起動時ではなく、必要になったタイミングでモデルを読み込むことで、起動時の負荷を軽減
- **入力チェック:** ファイル未選択や許可されていない拡張子のアップロードを検知し、トップページへ戻す
- **安全なファイル名処理:** secure_filename を使い、アップロードファイル名を安全な形式に変換
- **例外ログ:** モデル読み込み失敗時にはログへ詳細を出力し、原因調査をしやすくする

## Testing / CI

本プロジェクトでは、`pytest` と GitHub Actions を用いて、画像認識アプリの主要な Web フローを自動テストしています。

### テスト対象

現在のテストでは、以下を確認しています。

- トップページ `/` が正常に表示されること
- `/result` に画像なしで POST した場合、トップページへリダイレクトされること
- 許可されていない拡張子のファイルを POST した場合、トップページへリダイレクトされること
- 正常な画像をアップロードした場合、結果画面が表示されること
- 推論結果が結果画面に表示されること
- アップロード画像の URL が結果画面に含まれること

正常系のテストでは、本物の `image_classifier.h5` を読み込まず、`DummyModel` によって推論処理をモックしています。  
これにより、TensorFlow / Keras やモデルファイルの読み込みに依存せず、アップロード、前処理、結果表示までのアプリ側の処理を安定して検証できます。

### テスト実行

```bash
uv run pytest
```

## ディレクトリ構成

```text
.
├── app.py                 # Flask アプリケーション本体
├── train_cifar10.py        # モデル学習用スクリプト
├── karas.py                # 前処理関連の検証・補助スクリプト
├── image_classifier.h5     # 学習済みモデル
├── tests/
│   └── test_app.py         # pytest による Web フローのテスト
├── .github/
│   └── workflows/
│       └── ci.yml          # GitHub Actions の CI 設定
├── data/
│   └── uploads/            # アップロード画像の保存先
├── templates/
│   ├── index.html          # アップロード画面
│   ├── result.html         # 結果表示画面
│   └── layout.html         # 共通レイアウト
├── static/                 # CSS、画像、favicon などの静的ファイル
├── pyproject.toml          # 依存関係と pytest 設定
└── uv.lock                 # uv のロックファイル

```

- **CI/CD パイプラインの拡張**
  - 現在は GitHub Actions で pytest を自動実行
  - 今後はデプロイ前チェック、実モデル読み込み検証を検討

- **実モデルの検証強化**
  - 現在のテストでは `DummyModel` を使って Web フローを検証

- **アップロード処理のセキュリティ強化**
  - 拡張子チェックに加えて MIME タイプ検証を追加
  - ファイルサイズ制限を追加
  - 不正ファイルや破損画像へのエラーハンドリングを強化

- **フィードバック機能と MLOps ループの構築**
  - ユーザーのフィードバックを収集し、誤分類データを分析
  - 将来的に再学習やモデル改善のサイクルを構築

- **物体検出モデルへの拡張検討**
  - 現在は画像全体を 1 クラスに分類する画像分類モデル

- **FastAPI への移行検討**
  - 推論処理の分離や API 化を進める場合、FastAPI への移行を検討
  - 自動 API ドキュメントや非同期処理による拡張性向上を狙う
